git学习笔记

Git is a distribute control system.
Git is free software.

我学习git和github这份笔记是以廖雪峰教程为参考的

创建版本库，两点内容：

初始化一个Git仓库，使用git init命令。

添加文件到Git仓库，分两步：
.

  • 第一步，使用命令git add , (git add readme.txt)可以反复多次使用，

    添加多个文件；

    .

  • 第二步，使用命令git commit，(git commit -m “wrote a readme file”)

    完成,git commit命令执行成功后会告诉你，1个文件被改动。

要随时掌握工作区当前的状态：

使用git status命令。如果git status告诉你有文件被修改过，用git diff
($ git diff readme.txt )可以查看修改内容。cat readme.txt可以查看文档内容。

版本回退：

HEAD指向的版本就是当前版本，因此，Git允许我们在版本的历史之间穿梭，穿梭前，用git log可以查看提交历史，以便确定要回退到哪个版本，–pretty=oneline参数可以看只有版本号形式，再加上 –abbrev-commit可以看只有版本号前六位的(git log –pretty=oneline –abbrev-commit
)。使用命令git reset –hard commit_id($ git reset –hard HEAD^)。要重返未来，用git reflog查看命令历史，以便确定要回到未来的哪个版本。cat readme.txt可以查看文档内容。

工作区和暂存区（关于这一部分需要概念，多看几次,有截图~~)：

工作区(working Director)就是电脑里的目录，比如learngit文件夹;版本库(Repository)：工作区有一个隐藏目录.git，这个就是GIT的版本库。 前面讲了我们把文件往Git版本库里添加的时候，是分两步执行的：

  • 第一步是用git add把文件添加进去，实际上就是把文件修改添加到暂存区；
  • 第二步是用git commit提交更改，实际上就是把暂存区的所有内容提交到当前分支。
    因为我们创建Git版本库时，Git自动为我们创建了唯一一个master分支，
    所以，现在，git commit就是往master分支上提交更改。
    你可以简单理解为，需要提交的文件修改通通放到暂存区，然后，一次性提交暂存区的
    所有修改。

管理修改：

为什么Git比其他版本控制系统设计得优秀，因为Git跟踪并管理的是修改，而非文件。每次修改，如果不add到暂存区，那就不会加入到commit中。

撤销修改(慎重使用,我用了排的版都不见了，时光穿梭也没有用！！)：

  • 场景1：当你改乱了工作区某个文件的内容，想直接丢弃工作区的修改时，用命令git checkout – readme.txt。
  • 场景2：当你不但改乱了工作区某个文件的内容，还添加到了暂存区时，想丢弃修改，分两步，第一步用命令git reset HEAD file，就回到了工作区（场景1）。
  • 场景3：已经提交了不合适的修改到版本库时，想要撤销本次提交，参考版本回退一节，不过前提是没有推送到远程库。这部分记得看视频，有截图~~

删除文件：

rm test.txt如果add 可以从版本库恢复到工作区用 git checkout – test.txt

GIT是分布式版本控制系统，同一个GIT仓库，可以分布到不同的机器上。

远程仓库：
为什么GitHub需要SSH Key(SSH Key 在c/user/h/.ssh)呢？因为GitHub需要识别出你推送的提交确实是你推送的，而不是别人冒充的，而Git支持SSH协议，所以，GitHub只要知道了你的公钥，就可以确认只有你自己才能推送。当然，GitHub允许你添加多个Key。假定你有若干电脑，你一会儿在公司提交，一会儿在家里提交，只要把每台电脑的Key都添加到GitHub，就可以在每台电脑上往GitHub推送了。

添加远程库：

要关联一个远程库，使用命令git remote add origin git@server-name:path/repo-name.git；
关联后，使用命令git push -u origin master第一次推送master分支的所有内容,以后不需要 -u此后，每次本地提交后，只要有必要，就可以使用命令git push origin master推送最新修改；

克隆远程库：

要克隆一个仓库,首先必须知道仓库的地址(git clone git@github.com:zpdmm/gitskills.git)
gitskills是在远程库创建的新仓库，然后使用git clone命令克隆。Git支持多种协议，包括https，但通过ssh支持的原生git协议速度最快。

分支管理：

分支好神奇的 高效又安全 写出git的那个胖纸好腻害啊~~我们先创建分支dev在dev上工作，之后提交文件，再合并master分支，这样master就可以直接跟上最新的同步而之前不受任何影响，之后删除dev就可以了。

创建合并分支：

Git鼓励大量使用分支：

  • 查看分支：git branch
  • 创建分支：git branch dev
  • 切换分支：git checkout dev
  • 创建+切换分支：git checkout -b dev(等于前面两者的综合)
  • git checkout master
  • 合并某分支到当前分支：git merge dev
  • 删除分支：git branch -d dev

解决冲突：

在分支合并时由于两个分支都有提交，提交内容在同一行有相似内容就会出现冲突，导致无法合并，手动修改后就可以提交了。（有截图~~）。

分支管理策略(看了几个博客之后我晕了)：
在开发过程中使用no–ff（no–fastfoward）模式，这个要看廖雪峰的博客

Bug分支：

修复bug时，我们会通过创建新的bug分支（issue-001）进行修复，然后合并到master，最后删除；当手头工作没有完成时，先把工作现场（dev）git stash一下，git stash list可以看当前的stash然后去修复bug，修复后，再git stash pop，回到工作现场。

Feature分支（功能）：

添加一个新功能时，你肯定不希望因为一些实验性质的代码，把主分支搞乱了，所以，每添加一个新功能，最好新建一个feature分支，在上面开发，完成后，合并，最后，删除该feature分支。

在这我发现原来可以管理的不只是.txt文件(git add vulcan.py),我就在想咋管理代码.

标签管理(其实我不知道为啥要标签)

  • 命令git tag (git tag v1.0 )用于新建一个标签，默认为HEAD,也可以指定
    一个commit id(git tag v0.9 522a797)；
  • git tag -a v1.0 -m “blablabla…”可以指定标签信息；
  • git tag -s -m “blablabla…”可以用PGP签名标签；(这个没有实践)
  • 命令git tag可以查看所有标签。git show v1.0可以看具体信息”blalalala..”
  • 命令git push origin v1.0可以推送一个本地标签；
  • 命令git push origin –tags可以推送全部未推送过的本地标签；
  • 命令git tag -d v1.0可以删除一个本地标签；
  • 命令git push origin :refs/tags/可以删除一个远程标签。

使用github

终于到了这一节 本来我是只打算学会github的。。。

如何参与一个开源项目呢？比如人气极高的bootstrap项目，这是一个非常强大的CSS框架，你可以访问它的项目主页https://github.com/twbs/bootstrap，点“Fork”就在自己的账号下克隆了一个bootstrap仓库，然后，从自己的账号下clone：
git clone git@github.com:zpdmm/bootstrap.git
一定要从自己的账号下clone仓库，这样你才能推送修改。如果从bootstrap的作者的仓库地址git@github.com:twbs/bootstrap.git克隆，因为没有权限，你将不能推送修改。
